# ПроКомьюнити — полное приложение (бэкенд + Telegram Mini App)

Один сервис на **FastAPI** отдаёт и REST API, и само приложение. База — **SQLite**
(без отдельного сервера БД, заводится одной командой). Данные общие для всех
пользователей: сигналы плотности и прогресс подопечных синхронизируются между
устройствами. Авторизация — через подпись Telegram `initData`.

## Стек
- Backend: FastAPI, SQLAlchemy (async), SQLite (aiosqlite)
- Frontend: Telegram Mini App одним файлом (Leaflet-карта, Space Grotesk, кольцевой
  индикатор пути, анимированный дашборд) — раздаётся тем же сервисом
- Деплой: Docker (один контейнер)

## Запуск локально (2 минуты)
```bash
cd procommunity_full
./run.sh                 # ставит зависимости, создаёт БД с демо-данными, поднимает сервер
# открой http://localhost:8000
```
Вручную, если нужно:
```bash
pip install -r requirements.txt
python -m app.seed
uvicorn app.main:app --reload --port 8000
```
В браузере приложение работает в демо-режиме (без Telegram показывается демо-новичок).
Переключатель ролей вверху — для показа жюри.

## Деплой (чтобы открыть в Telegram)
Нужен публичный HTTPS-URL. Самый быстрый путь — **Render** или **Railway**:
1. Залей папку в GitHub-репозиторий.
2. На render.com / railway.app → New → Deploy from repo. Dockerfile определится сам.
3. Получишь адрес вида `https://…onrender.com`.
4. Задай `BOT_TOKEN` в переменных окружения сервиса (иначе Telegram-подпись не
   проверяется — для демо можно оставить пустым, тогда работает демо-режим).
5. У **@BotFather**: `/mybots` → бот → *Bot Settings* → *Menu Button* → вставь URL.
   Открой бота, нажми кнопку меню — приложение откроется внутри Telegram.

Любой VPS / Yandex Cloud:
```bash
docker build -t procommunity . && docker run -p 8000:8000 -e BOT_TOKEN=xxxx procommunity
```
(за nginx с сертификатом, либо через встроенный HTTPS платформы).

## Переменные окружения (.env)
- `BOT_TOKEN` — токен бота от @BotFather. Нужен для проверки подписи Telegram
  initData в проде. Пусто → демо-режим (открывается в браузере).
- `ENV` — `dev` (демо-пользователь разрешён) или `prod`.
- `FEE_SAVED` — ставка сэкономленного реф-бонуса для панели партнёра (22000 ₽).
- `DB_PATH` — путь к файлу SQLite (по умолчанию `procommunity.db`).

## API (кратко)
- `GET  /api/snapshot` — всё для фронта одним запросом (me, mentees, density, requests, stats)
- `POST /api/me/checklist` `{step}` — отметить пункт своего чек-листа
- `POST /api/mentee/{id}/checklist` `{step}` · `POST /api/mentee/{id}/flag` `{flag,value}`
- `POST /api/mentee` · `POST /api/density` · `POST /api/partners`
- `GET  /api/health`
Swagger: `/docs`.

## Прод-версия для сбора фидбэка (жюри видит всё)

В приложении есть кнопка-конверт (вверху справа) — отзыв со звёздами и текстом.
Отзыв сохраняется в базу И отправляется в закрытый Telegram-канал через бота,
поэтому переживает передеплой. В роли **«Партнёр»** есть панель «Отзывы» —
жюри видит все отзывы прямо в приложении.

### Переменные окружения на Render (Environment)
- `BOT_TOKEN` — токен бота от @BotFather (реальные имена пользователей + отправка отзывов).
- `ENV` = `prod`.
- `FEEDBACK_CHAT_ID` — id закрытого канала/группы, куда падают отзывы.

### Как получить FEEDBACK_CHAT_ID
1. Создай закрытый Telegram-канал или группу «Отзывы ПроКомьюнити».
2. Добавь туда своего бота **как администратора**.
3. Узнай id канала: перешли любое сообщение из канала боту **@userinfobot**
   (он покажет id вида `-1001234567890`), либо открой
   `https://api.telegram.org/bot<ТОКЕН>/getUpdates` после поста в канал.
4. Впиши этот id в `FEEDBACK_CHAT_ID` на Render → сервис передеплоится.
5. **Добавь жюри в этот канал** — и они видят каждый отзыв в реальном времени.

### Чтобы не «засыпал» на защите
Free-инстанс Render засыпает после простоя (первый запрос потом ждёт ~50 сек).
Заведи бесплатный пинг на **cron-job.org** или **UptimeRobot**, который раз в
10 минут дёргает `https://<твой-адрес>/api/health`. Либо просто открой ссылку
за пару минут до показа, чтобы сервис проснулся.

### Важно про данные (честно)
SQLite на Render **эфемерна**: при передеплое отзывы и точки в базе обнуляются
до демо-данных. Именно поэтому отзывы дублируются в Telegram-канал — там они
не пропадают. Если нужно, чтобы ВСЕ данные жили постоянно (не только отзывы),
подключи бесплатный PostgreSQL (Neon / Supabase) и задай `DATABASE_URL` —
модели уже на SQLAlchemy, менять почти нечего. Под дедлайн достаточно
Telegram-канала для отзывов.

### Как жюри увидит всё
- Ссылка/бот открывают приложение целиком; переключатель ролей вверху — все три
  стороны (Новичок / Наставник / Партнёр).
- Панель «Отзывы» в роли «Партнёр» — собранный фидбэк внутри приложения.
- Плюс сам Telegram-канал отзывов, если добавить туда жюри.

## UX-обновление: тур, языки, соглашение

- **Онбординг-тур.** При первом входе (после согласия) — пошаговое объяснение
  интерфейса: переключатель ролей → «Путь» → «Плотность» → «Напарник» → отзыв.
  Повторить можно кнопкой «?» рядом с подписью ролей.
- **Языки.** Кнопка-глобус слева вверху: Русский, Oʻzbekcha, Тоҷикӣ, العربية, English.
  Арабский — с RTL. Выбор языка запоминается. Язык можно выбрать и на экране согласия.
- **Пользовательское соглашение + согласие.** Первый экран: приветствие, ссылка
  «Читать соглашение» (полный текст), чекбокс и «Принять и продолжить». Текст ToS
  включает дисклеймер «независимый пилот, не официальный продукт Яндекса», раздел
  о геоданных (только по согласию) и о самозанятости. **Впиши свой @-контакт**
  в конце текста ToS (переменная `TOS_HTML` в `static/index.html`).

### Важные оговорки
- **Переводы uz / tg / ar — черновые (машинного качества).** Перед тем как
  показывать носителям/жюри, дай native-спикеру вычитать, особенно таджикский и
  арабский. Правятся в объекте `I18N` вверху скрипта.
- Локализованы навигация, заголовки, тур и соглашение. Несколько строк с числами
  (подзаголовок наставника, статусы, пояснение экономики, плейсхолдеры форм) пока
  на русском — расширяется добавлением ключей в `I18N` и заменой строки на `t('key')`.
- Согласие и тур показываются один раз (хранятся в localStorage устройства).
  Чтобы снова увидеть экран согласия на защите — открой в режиме инкогнито/новом
  браузере; тур повторяется кнопкой «?».

### Про интеграцию с Яндекс-аккаунтом
Осознанно НЕ реализована: Telegram Mini App уже даёт личность через initData,
OAuth Яндекса в вебвью работает криво, а сверка статуса курьера требует доступа к
API Яндекс Про (только по партнёрству). Для жюри — как пункт роадмапа: «в проде —
SSO через Яндекс ID и сверка статуса в Яндекс Про по партнёрству».

## Боевой прод: постоянные данные + метрики для презентации

Добавлены: постоянная база (PostgreSQL), логирование реальных событий и
дашборд метрик на `/admin`.

### 1. Постоянная база (обязательно для реальных данных)
SQLite на Render стирается при передеплое. Для боевого прода подключи бесплатный
PostgreSQL:
- **Render PostgreSQL** (проще всего, та же платформа): создай инстанс, скопируй
  *Internal Database URL*, добавь его в веб-сервис как переменную `DATABASE_URL`.
- Или **Neon** / **Supabase** (постоянный free tier): скопируй connection string
  (вида `postgresql://...?sslmode=require`) в `DATABASE_URL`.
Код сам приводит URL к async-драйверу и включает SSL для Neon/Supabase. Таблицы
создаются автоматически при старте; демо-данные подсеваются, если база пустая.

### 2. Метрики (дашборд для презентации)
- Задай переменную `ADMIN_KEY` = длинная случайная строка.
- Открой `https://<твой-адрес>/admin?key=<ADMIN_KEY>` — увидишь дашборд:
  уникальные пользователи, открытия, шаги чек-листа, сигналы плотности, заведённые
  новички, отзывы + средняя оценка, открытия по дням.
- Ключ гейтит только данные (`/api/metrics`); саму страницу не показывай публично.

### 3. Keep-alive (иначе первый заход = 50 сек)
Для реальных пользователей сервис не должен засыпать: заведи бесплатный пинг на
**cron-job.org** / **UptimeRobot** на `https://<адрес>/api/health` раз в 5–10 минут.

### 4. Как получить реальных пользователей
- Посев в курьерских районных чатах (ты сам в них есть) со ссылкой на бота.
- QR-коды в точках выдачи заказов и пунктах аренды.
- Реферальная механика «приведи друга».
- Несколько знакомых опытных курьеров — как первые наставники.

### Роли в проде
Переключатель ролей сейчас — демо-режим (удобно показывать все три стороны). Для
чистого пилота логика такая: пришедший курьер по умолчанию «новичок»; опытных
вручную помечаешь наставниками; вид «Партнёр»/метрики — только у тебя по ключу.
Полноценное разграничение ролей по личности из initData — следующий шаг.

### Честно про метрики на презентации
Retention (удержание) — метрика 30 дней и с контрольной группой; за несколько дней
до защиты валидной цифры не будет. Показывай **раннюю тягу**: реальные открытия,
прохождение чек-листа, сигналы, отзывы, DAU по дням — из дашборда `/admin`. Про
удержание говори честно: «пилот запущен, ридаут через 30 дней». Это сильнее и
не разваливается на первом вопросе жюри.

## Реальный режим: профили, роли, город (SHOWCASE=false)

По умолчанию `SHOWCASE=true` — демо для жюри (переключатель ролей, сид-данные).
Чтобы включить реальную экосистему, задай переменную `SHOWCASE=false`.

**Что происходит в реальном режиме:**
- **Онбординг.** Первый вход → экран «Кто ты?» (Новичок / Наставник) → выбор
  города (кнопка «Определить автоматически» через геолокацию + reverse-geocode,
  или ручной список из 12 городов). Роль и город пишутся в профиль (`users`).
- **Фильтрация по городу на бэкенде.** Каждый видит плотность и карточки
  «напарник» ТОЛЬКО своего города; карта центрируется на городе. Чужие города
  не показываются.
- **Матч наставник↔новичок.** Наставник видит очередь новичков своего города и
  берёт подопечного кнопкой; новичку назначается наставник.
- **Профиль-вкладка** со статистикой (новичок: день/прогресс/наставник; наставник:
  подопечные/удержано).
- Переключатель ролей скрыт (роль берётся из базы).

**Критично при первом деплое реального режима:** новые колонки/таблицы
(`users`, `city` у density/partner) не появятся в старой базе сами. Сбрось таблицы
в Postgres (данных для прода там нет):
```
DROP TABLE IF EXISTS users, density_reports, partner_requests, participants, mentors, feedback, events CASCADE;
```
При старте создадутся заново с правильной схемой.

**Оговорки:**
- Reverse-geocoding — через публичный Nominatim (OSM) с клиента: есть лимиты,
  для прода лучше платный геокодер (Яндекс/Google). Ручной список городов —
  надёжный фолбэк.
- GPS не блокирует приложение: если доступ не дан — мягкий баннер, всё остальное
  работает.
- Интерфейс реального режима переведён на русский и английский; узб/тадж/араб
  для новых экранов пока откатываются на русский (заполняется в объекте `I18N3`).

## Умная карта плотности: зоны + тепловая карта

Карта показывает НЕ отдельных курьеров, а агрегированную плотность по зонам —
это приватно (никакой слежки) и осмысленно.

- **Автопривязка к зоне.** Каждый сигнал бэкенд относит к ячейке сетки
  ~600 м (`ZONE_CELL_DEG`).
- **Агрегация на бэкенде.** По зоне считается: оценка числа курьеров (макс. из
  свежих сигналов), среднее ожидание, число сигналов. В реальном режиме —
  только свежие сигналы (`DENSITY_TTL_MIN`, по умолч. 60 мин) и только своего города.
- **Тепловая карта** (Leaflet.heat): зелёный → жёлтый → красный по плотности зоны;
  тап по зоне — детали. Если плагин не загрузился — фолбэк на цветные круги-зоны.
- Список под картой — недавние именованные сигналы (для читаемости).

## Полишинг карты + живой поллинг

- **Тепловая карта** доведена: мягкий градиент зелёный→жёлтый→красный, аккуратные
  точки-зоны для тапа, стилизованные под бренд контролы зума и попапы.
- **«Вживую».** На карте — индикатор с пульсирующей точкой. Данные плотности
  обновляются каждые **25 секунд** без перезагрузки: обновляются только слои карты
  и список, экран не дёргается.
- Поллинг работает **только на вкладке «Плотность»**, ставится на паузу, когда
  приложение свёрнуто (`document.hidden`) или открыт лист; при возврате в приложение
  карта обновляется сразу. Частота — константа `25000` мс в `startPolling`.

## Матчинг по городу + инфраструктура

**Матч наставник↔новичок:**
- При выборе города новичок **авто-назначается** наименее загруженному наставнику
  своего города (мгновенная поддержка). Если свободных нет — уходит в очередь.
- Лимит активных подопечных на наставника — `MENTOR_MAX_MENTEES` (по умолч. 8).
- Наставник берёт из очереди (`/take`) и может **открепить** (`/release`) — новичок
  возвращается в очередь и может быть подхвачен другим.
- При матче обоим уходит уведомление от бота (если пользователь запускал бота).

**Инфраструктура под нагрузку:**
- **Индексы**: композитные под горячие запросы — `(city, created_ms)` для плотности
  и карточек, `(role, mentor_id)` и `(role, city)` для пользователей. Создаются при
  (пере)создании таблиц.
- **Рейт-лимиты** (in-memory) на запись: плотность 15/мин, карточки 6/10мин,
  отзывы 5/10мин, роль/город 10/мин, трекинг 60/мин. Превышение → HTTP 429.
  Работает при одном воркере (Render free, `WEB_CONCURRENCY=1`); при масштабировании
  на несколько воркеров нужен Redis.
- **Очистка старья**: фоновая задача раз в `CLEANUP_EVERY_MIN` мин удаляет протухшую
  плотность (>2×TTL), события старше 90 дней, карточки старше 14 дней.

## Миграции базы (Alembic) — изменения схемы без потери данных

Схема БД теперь под управлением **Alembic**. Миграции применяются автоматически
при каждом деплое: старт-команда сначала гоняет `alembic upgrade head`, затем
поднимает uvicorn (см. Dockerfile / run.sh). `create_all` больше не используется.

### Первый переход на миграции (одноразово)
Текущая база в проде была создана старым способом и не знает про Alembic, поэтому
её нужно один раз обнулить (данные там демо-шные). В консоли Postgres (Neon / Render):
```sql
DROP TABLE IF EXISTS users, density_reports, partner_requests, participants,
                     mentors, feedback, events, alembic_version CASCADE;
```
Затем задеплой этот код и **Manual Deploy → Clear build cache & deploy**. На старте
`alembic upgrade head` создаст схему заново (уже с `city`, `users`, индексами) и
проставит версию. Это чинит падение `column density_reports.city does not exist`.

### Как менять схему дальше (уже с реальными данными — без потерь)
1. Меняешь модель в `app/models.py` (добавляешь поле/таблицу).
2. Локально генерируешь миграцию:
   ```bash
   alembic revision --autogenerate -m "что изменил"
   ```
3. **Открываешь** сгенерированный файл в `alembic/versions/` и проверяешь глазами
   (autogenerate не всегда идеален).
4. Локально проверяешь: `alembic upgrade head` (на SQLite).
5. Коммитишь миграцию вместе с кодом и деплоишь. На проде `alembic upgrade head`
   применит её к существующей базе — **старые данные останутся**, добавится только
   новое (проверено: строка сохраняется, колонка добавляется).

### Важный нюанс при добавлении колонок в непустую таблицу
Новую колонку делай **nullable** (`Mapped[str | None]`) или задавай `server_default`.
Колонка `NOT NULL` без `server_default` не добавится к таблице, где уже есть строки
(это ограничение любой СУБД, не Alembic).

### Локальный запуск
```bash
alembic upgrade head    # создать/обновить схему
```

## Ручной выбор точки сигнала (перекрестие)

Кнопка «+ Оставить сигнал» теперь открывает режим выбора места: карта
разворачивается, в центре — перекрестие (навигационный прицел). Пользователь
двигает карту так, чтобы нужное место было под прицелом, жмёт «Здесь» — и только
потом заполняет данные (кол-во курьеров, ожидание). Точка = центр карты в момент
подтверждения. Кнопка «по геолокации» в форме остаётся как альтернатива.

## Точные точки сигналов + удаление автором

- Сигналы теперь рисуются на карте в **точных координатах** выбранной точки (свои —
  с тёмной обводкой), тепловые зоны остались фоновой подложкой. Больше нет смещения
  «в центр ячейки».
- Тап по сигналу открывает карточку. Свой сигнал можно **«Ещё актуально»** (продлить
  TTL) или **«Удалить»**. Чужие — только просмотр.
- Права строгие: сигнал привязан к автору (`author_id` = telegram_id), удалить/продлить
  может только он (проверено на бэкенде). Сигналы, созданные до этой правки, автора не
  знают — их не удалить кнопкой (только по TTL).
- Схема расширена миграцией `density_reports.author_id` (nullable) — старые данные
  сохраняются.

## Интерактивный тур (оба режима) + демо для жюри

- **Тур работает и в демо, и в проде.** Запускается при первом входе (после согласия
  и — в проде — после онбординга) и в любой момент по кнопке «?» в шапке (видна всегда).
- **Демо-режим (SHOWCASE=true)** — тур-презентация для жюри: сам проводит через путь
  новичка → карту плотности → поиск напарника → вид наставника → B2B-панель → отзыв.
  Переключатель ролей вверху позволяет вручную смотреть за новичка и за наставника.
- **Прод-режим (SHOWCASE=false)** — тур по реальным вкладкам под роль пользователя
  (новичок или наставник): путь/подопечные → плотность → напарник → профиль → отзыв.
- Тексты тура на русском и английском (остальные языки — фолбэк на русский).

## Демо = прод + переключатель ролей

Теперь демо-режим (SHOWCASE=true) использует ТУ ЖЕ реальную механику, что и прод:
онбординг, определение города по GPS/вручную, карта по своему городу, живые сигналы.
Единственное отличие — вверху **переключатель Новичок / Наставник / Бизнес**, которым
жюри легко смотрит продукт под каждой ролью.

- Чтобы обе роли были «живыми» в городе гостя, демо **досеивает** данные в его город:
  новичку — наставника, наставнику — 3 подопечных с разным прогрессом, на карту —
  несколько сигналов (всё только при SHOWCASE=true; прод остаётся чистым).
- «Новичок»/«Наставник» реально переключают роль (POST /api/role) и перезагружают
  данные — это не картинка, а настоящий опыт роли.
- «Бизнес» показывает B2B-дашборд (удержание и экономия) с наглядными числами — для
  питча жюри.
- Прод (SHOWCASE=false): переключателя нет, всё по-настоящему под роль пользователя.
- Тур работает в обоих режимах; в демо первый шаг подсвечивает переключатель.

## Карта: чистые точки, кнопка наставника в демо, метрики Бизнеса

- **Карта плотности переработана:** одиночный сигнал — чёткая точка + мягкий цветной
  ореол (вектор, идеально держится за точку при зуме/сдвиге). Тепловая подложка
  остаётся только там, где скучкованы 2+ сигнала. Тепло перерисовывается при зуме —
  ничего не «отваливается».
- **Демо-наставник:** кнопка «Написать» в демо-режиме никуда не ведёт (показывает
  пояснение). В реальном режиме, где наставник настоящий, ссылка на Telegram рабочая.
- **Метрики Бизнеса приведены к согласованности:** верхнее число показывает реальный
  прирост в п.п. (а не 0), а деньги = «удержано сверх контроля» × реф-бонус. Демо-цифры:
  9/12 (75%) против 6/12 (50%), +25 п.п., ≈ 66 000 ₽. Всё бьётся между собой.

## Рассылка всем пользователям (broadcast)

На странице `/admin` (доступ по `ADMIN_KEY`) появился блок «Рассылка всем
пользователям»: поле для текста + кнопка «Отправить всем». Сообщение уходит всем
из таблицы `users` (кто заходил в приложение), пачками по 25 с паузой 1 сек — под
лимит Telegram (~30 сообщений/сек). Прогресс показывается вживую.

Важно (ограничения Telegram):
- Бот может писать только тем, кто **сам запускал бота** (нажимал Start / открыл
  через бота). Кто открыл Mini App, но не запускал бота, — не получит (это правило
  Telegram, не баг). В статусе видно «доставлено N, не дошло M».
- Не злоупотребляйте частотой и рекламой — за спам Telegram может ограничить бота.

Также доступно программно:
`POST /api/broadcast?key=ADMIN_KEY` с телом `{"text":"...ссылка на форму..."}`
→ `{"ok":true,"targets":N}`. Статус: `GET /api/broadcast/status?key=ADMIN_KEY`.
